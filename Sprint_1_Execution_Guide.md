# الدليل التفصيلي لتنفيذ مهام السبرنت الأول (Sprint 1)

هذا الدليل مخصص لشرح الخطوات الدقيقة لكل مهمة في السبرنت الأول لمنصة Tendex AI، لضمان التنفيذ الخالي من الأخطاء والالتزام التام بمعايير الجودة المحددة في وثائق المشروع.

## تعليمات عامة قبل البدء بأي مهمة:
1. **تحديث المستودع:** دائماً ابدأ بعمل `git pull` للتأكد من أنك تعمل على أحدث نسخة من الكود.
2. **قراءة الوثائق:** راجع الوثائق المرجعية المذكورة في كل مهمة (مثل `Architecture_v1` أو `Code_Quality`).
3. **تحديث سجل التقدم:** بعد الانتهاء من أي مهمة، **يجب** تحديث ملف `PROGRESS.md`.
4. **الرفع إلى GitHub:** استخدم رسائل Commit واضحة باللغة الإنجليزية (مثل `feat: setup docker compose`).

### TASK-101: تهيئة مستودع المشروع وهيكلة المجلدات
**الهدف:** إنشاء الهيكل الأساسي للمشروع (Monorepo).

**ما يجب كتابته (الملفات المطلوبة):**
- إنشاء المجلدات التالية في الجذر: `frontend/`, `backend/`, `infrastructure/`, `docs/`.
- إنشاء ملف `.gitignore` شامل في الجذر يتضمن استثناءات لـ Node.js (مثل `node_modules`) و .NET (مثل `bin/`, `obj/`).
- إنشاء ملف `README.md` مبدئي يشرح كيفية تشغيل المشروع محلياً.

**ما يجب إرفاقه/تسليمه:**
- رفع الهيكلة الجديدة إلى GitHub.
- تحديث ملف `PROGRESS.md` بإضافة إنجاز TASK-101.

**أخطاء شائعة يجب تجنبها (المخاطر):**
- ❌ نسيان إضافة `.gitignore` مما يؤدي إلى رفع ملفات ضخمة وغير ضرورية (مثل `node_modules`) إلى المستودع، مما يبطئ العمل.
- ❌ استخدام أسماء مجلدات باللغة العربية أو بأحرف كبيرة (يجب استخدام أحرف صغيرة باللغة الإنجليزية لضمان التوافق مع أنظمة Linux).

### TASK-102: إعداد بيئة Docker و Docker Compose
**الهدف:** تجهيز البنية التحتية المحلية باستخدام الحاويات.

**ما يجب كتابته (الملفات المطلوبة):**
- إنشاء ملف `docker-compose.yml` داخل مجلد `infrastructure/` (أو في الجذر).
- تعريف الخدمات التالية:
  - `sqlserver`: صورة SQL Server 2022.
  - `redis`: صورة Redis 7.
  - `rabbitmq`: صورة RabbitMQ.
  - `minio`: صورة MinIO.
  - `qdrant`: صورة Qdrant.
- إعداد الـ Volumes لكل خدمة لضمان عدم فقدان البيانات عند إيقاف الحاويات.
- إعداد ملف `.env.example` يحتوي على متغيرات البيئة الأساسية (مثل كلمات المرور الافتراضية للتطوير المحلي).

**ما يجب إرفاقه/تسليمه:**
- ملف `docker-compose.yml` وملف `.env.example`.
- تحديث `PROGRESS.md`.

**أخطاء شائعة يجب تجنبها (المخاطر):**
- ❌ كتابة كلمات مرور حقيقية أو حساسة (مثل الخاصة بخوادم Hostinger) في ملف `docker-compose.yml`. يجب استخدام متغيرات البيئة دائماً.
- ❌ عدم تحديد Volumes، مما يؤدي إلى مسح قواعد البيانات بالكامل بمجرد إعادة تشغيل الحاوية.

### TASK-103: تهيئة مشروع الواجهة الخلفية (.NET 10)
**الهدف:** بناء هيكل Clean Architecture للواجهة الخلفية.

**ما يجب كتابته (الملفات المطلوبة):**
- داخل مجلد `backend/`، إنشاء Solution جديد لـ .NET 10.
- إنشاء 4 مشاريع (Projects) تمثل طبقات Clean Architecture:
  1. `TendexAI.Domain` (Class Library)
  2. `TendexAI.Application` (Class Library)
  3. `TendexAI.Infrastructure` (Class Library)
  4. `TendexAI.API` (Web API / Minimal APIs)
- إعداد المراجع (References) بين المشاريع بشكل صحيح (API يعتمد على Application و Infrastructure، و Application يعتمد على Domain).

**ما يجب إرفاقه/تسليمه:**
- ملفات الـ Solution (`.sln`) والمشاريع (`.csproj`).
- تحديث `PROGRESS.md`.

**أخطاء شائعة يجب تجنبها (المخاطر):**
- ❌ إضافة اعتماديات خارجية (مثل Entity Framework أو مكتبات قاعدة البيانات) في طبقة `Domain`. طبقة الـ Domain يجب أن تكون نقية تماماً ولا تعتمد على أي شيء.
- ❌ استخدام Controllers التقليدية بدلاً من Minimal APIs (المتطلبات تنص صراحة على استخدام Minimal APIs لضمان الأداء العالي).

### TASK-104: تهيئة مشروع الواجهة الأمامية (Vue.js 3)
**الهدف:** إعداد مشروع الواجهة الأمامية بالتقنيات المعتمدة.

**ما يجب كتابته (الملفات المطلوبة):**
- داخل مجلد `frontend/`، تهيئة مشروع Vue 3 باستخدام Vite و TypeScript.
- تثبيت وإعداد Tailwind CSS v4.
- تثبيت وإعداد PrimeVue 4.
- تثبيت Pinia لإدارة الحالة (State Management).
- إعداد ملف `tailwind.config.js` (أو إعدادات v4) لدعم الخصائص المنطقية (RTL).

**ما يجب إرفاقه/تسليمه:**
- ملفات مشروع Vue (مثل `package.json`, `vite.config.ts`, `main.ts`).
- تحديث `PROGRESS.md`.

**أخطاء شائعة يجب تجنبها (المخاطر):**
- ❌ استخدام هوامش ثابتة الاتجاه في Tailwind (مثل `ml-4` أو `pr-2`). يجب استخدام الخصائص المنطقية حصراً (`ms-4`, `pe-2`) لضمان عمل الواجهة بشكل صحيح عند التبديل بين اللغتين العربية والإنجليزية.
- ❌ استخدام JavaScript العادي بدلاً من TypeScript.

### TASK-105: إعداد قاعدة البيانات المركزية (master_platform)
**الهدف:** إعداد Entity Framework Core وتشفير البيانات.

**ما يجب كتابته (الملفات المطلوبة):**
- في طبقة `Infrastructure` (الواجهة الخلفية)، إعداد `DbContext` الخاص بقاعدة البيانات المركزية.
- تكوين الاتصال بـ SQL Server.
- إعداد نموذج Database-per-Tenant (تجهيز البنية التحتية لدعم قواعد بيانات متعددة لاحقاً).
- تفعيل ميزة Ledger Tables لبعض الجداول الحساسة (إن أمكن في هذه المرحلة).

**ما يجب إرفاقه/تسليمه:**
- ملفات `DbContext` و `Migrations` الأولية.
- تحديث `PROGRESS.md`.

**أخطاء شائعة يجب تجنبها (المخاطر):**
- ❌ تفعيل الحذف المتسلسل (Cascade Delete). يجب استخدام `DeleteBehavior.NoAction` أو `Restrict` صراحة في الـ Fluent API لمنع حذف البيانات المرتبطة بالخطأ.
- ❌ تخزين نص الاتصال (Connection String) بشكل ثابت في الكود. يجب جلبه من `appsettings.json` أو متغيرات البيئة.

### TASK-106: إعداد مسارات التكامل المستمر (CI Pipelines)
**الهدف:** أتمتة عمليات البناء والاختبار.

**ما يجب كتابته (الملفات المطلوبة):**
- إنشاء مجلد `.github/workflows/` في جذر المشروع.
- إنشاء ملف `ci-backend.yml` لإعداد مسار بناء واختبار مشروع .NET عند كل Pull Request.
- إنشاء ملف `ci-frontend.yml` لإعداد مسار بناء واختبار مشروع Vue.js.

**ما يجب إرفاقه/تسليمه:**
- ملفات الـ YAML الخاصة بـ GitHub Actions.
- تحديث `PROGRESS.md`.

**أخطاء شائعة يجب تجنبها (المخاطر):**
- ❌ إعداد النشر المباشر (CD) إلى خوادم Hostinger في هذه المرحلة. النشر المباشر مخصص لـ Sprint 7. هذه المهمة تقتصر على الـ CI (البناء والاختبار التلقائي فقط) لضمان جودة الكود قبل دمجه.

## آلية التسليم النهائية لكل مهمة (Workflow Mechanism):
عندما يقوم الوكيل بتنفيذ أي مهمة، يجب عليه اتباع هذا التسلسل بدقة:

4. تنفيذ أوامر Git لرفع التغييرات:
   ```bash
   git add .
   git commit -m "feat: [وصف مختصر بالإنجليزية]"
   git push origin main
   ```
